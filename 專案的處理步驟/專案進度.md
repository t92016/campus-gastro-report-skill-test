## 踩坑記錄

### 坑 12：HTTP POST 批次填表會產生幽靈資料（案件編號空白）
- **原因**：多個 checkbox 值應在同一個 POST body 中以多個 `entry.XXX` 參數送出
- **解法**：修正 PowerShell 腳本，`&entry.XXX=值1&entry.XXX=值2`

### 坑 13：Google 表單 `moveItem` 只改變 UI 順序，不改變試算表欄位落點
- **原因**：`form.moveItem()` 只移動表單畫面順序，試算表中新欄位永遠被加到最後面
- **解法**：寫 `removeDuplicateHeightWeight` 或重建成全新題目

### 坑 14：表單連結的試算表欄位無法刪除
- **原因**：Google 表單連結的試算表欄位受保護
- **解法**：改為隱藏欄位 + 改名為 `_舊欄位_已隱藏`

### 坑 15：`form.setDestination()`（relinkFormToSheet）會建立新的回應工作表
- **症狀**：試算表出現兩個回應工作表（彙整總表 + 表單回覆 2），系統錯亂
- **原因**：`setDestination` 被 Google 視為「重新連結」，會建立新的回應表
- **解法**：寫 `switchToCorrectResponseSheet` 刪除舊表、把新表改名為彙整總表，再執行 `setupSpreadsheet`

### 坑 16：刪除表單題目後重建，新題目的試算表連結會遺失
- **症狀**：表單有身高體重題目，但送出後試算表沒出現對應欄位。管理者後台報錯 `COL.HEIGHT` 為 null
- **原因**：`removeDuplicateHeightWeight` 刪除了舊題目，`addHeightWeightToForm` 加了新題目，但新題目的試算表欄位連結未建立（Google 只在「首次有資料寫入」時才建立欄位，若已有同名欄位則可能衝突）
- **解法**：寫 `rebuildHeightWeightQuestions` 徹底刪除舊題目再重建（取得全新 item ID），搭配 `hideOldHeightWeightColumns` 確保舊殘留欄位不干擾。管理者後台加入 `fieldColMap[key]` 的 null 防呆

## 重大技術決策與理由

| 決策 | 理由 |
|------|------|
| 永遠不手動 `form.setDestination()` | 會建立重複的回應工作表，破壞系統 |
| 新增表單欄位：刪舊 → 重建（全新 ID） | 確保試算表連結正確建立 |
| 管理者後台為唯一資料修改入口 | 避免手動動試算表造成系統欄位錯亂 |
| 試算表重複欄位用隱藏而非刪除 | Google 不允許刪除表單連結的欄位 |
